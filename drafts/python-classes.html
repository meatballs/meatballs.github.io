Using Python Classes

Date Fri 26 October 2018 Tags python

Python is by no means the first programming language I learned - it's not even the first object oriented (OO) language - and that means I brought all the habits I'd acquired from other languages when I first started to learn it.

The OO language where I had most experience prior to Python was Java and, in Java, everything is a class. I'd become so used to that idea that I brought the habit with me into Python and it took me some time to realise that there are often better alternatives available.

Even once I'd learned about the various alternatives at my disposal, I still found it tricky to recognise when they might apply and how to go about choosing between them.

I suspect I'm not the only person facing this challenge, so I decided to write up what I've found to be helpful so far.

Definitions

First, let's agree what a class actually is! There are many definitions to be found in the literature, but here's mine:

A class is a template for creating objects.

Yes, in Python, a class is also an object in its own right, but its purpose is to act as a template for other objects.

But that definition leads to a question - what exactly is an object? Again, there are many definitions to be found, but here's what I shall use in this article:

An object is a set of data and code in a computer's memory which represents the state and behaviour of something that exists in the real world.

And now, we need to define what we mean by 'state' and 'behaviour.' The Oxford English Dictionary's definition of state says that is is:

"Condition in which a thing is, mode of existence as determined by circumstances"

So, an object's state defines its condition at a point in time and is liable to change over time.

Finally, we come to behaviour which the Oxford English Dictionary defines as:

"The way in which something acts or works"

For the purposes of this article, I'm going to define two types of behaviour: transitive and intransitive:

Transitive Behaviour: Behaviour which alters the state of the object

Intransitive Behaviour: Behaviour which has no effect on the state of the object (although it may effect some other external object)

Immutable Objects

Sometimes, it's obvious that the state of an object is never going to change once it's been created - perhaps the real-world entity that it represents can never change. e.g. An object to represent the transfer of a sum of money from one bank account to another will never change.

In other cases, even though the real-world entity might change, we have no intention of representing that change within our code. e.g. If I change a tyre on my car, the car has changed state but if my code is purely to monitor the miles per gallon of fuel I achieve, I'm probably not interested in that change within my code.

No Transitive Behaviour

A clue when considering existing code is whether the methods of a class represent transitive or intransitive behaviour. If there are no transitive methods, then either the object is immutable or its state is changed directly (by updating its attributes) without the use of a method.
